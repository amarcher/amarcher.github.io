Rails

Sunday, September 28, 2014

We've covered a number of topics in Phase 0, but we haven't touched on Rails yet. To get my feet wet, I went through a tutorial on Rails this week. I thought I'd give a brief overview of what it Rails is and what it lets you do.

What is it?

Rails is a "web application framework." It is a utility that lets you create the common components of your web app -- e.g. all the structures (models, views, controllers), files, and directories -- using terminal commands. As it creates these entities, it adds to them all the boilerplate code they'll use and defines implicit relationships between them.

Why is it useful?

When you're developing, you want to use your time defining how your app works -- i.e. building the unique parts of your application that let it do what it should. You don't want to spend your time writing the scaffolding of the app or the redundant boilerplate that's necessary for each entity you add to it. In fact these developer preferences are reflected in Rail's mission statement: (1) convention over configuration and (2) don't repeat yourself. This means: sacrifice some ability to customize your app infrastructure -- instead adhering to conventional setup -- and profit from not having to write code many times over.

How do you use it?

After installing the rails gem (from terminal: gem install rails) you can create your web app's scaffold using just one terminal command: rails new your_app_name. This creates a directory structure for your app and fills it with all the files/templates necessary to start serving your app locally with a simple command: rails server.

Rails applies the 'model view controller' framework for web apps. It lets you quickly "generate" new entites: models that will often have entries in your app's database, controllers that will let users interact with models, and views that will render your app to users via elegant templates. You use the rails generate your_entity its_name its_instantiation_arguments to do this. Significantly, as each of these entities is generated, all of the interfaces to it are defined according to convention. For instance, after you generate a controller, you can quickly configure its interface -- again with a simple terminal command -- to listen to conventional URI patterns and translate them to 'actions' which you can define as methods to respond to in your controller.

What does Ruby on Rails mean? Where does Ruby factor in?

You'll use the Ruby language in all the templates Rails created for you in order build out the classes that constitute your models and their methods that will be called as controllers. You can also use a templating language called eRuby to write code for templates that will render your app's interfaces as views.

Summary

From what I can tell, Rails looks like a powerful development framework. It seems like it will make building web apps significantly less redundant by abstracting some of the low-level configuration work. Can't wait to get started with it in about a week at DBC!
